# Contributing to db-o11y

This document covers best practices for contributing changes to this project.

---

## Branching

Always create a feature or fix branch before making changes. Never commit directly to `main`.

```bash
git checkout main
git pull
git checkout -b feature/<short-description>
# or
git checkout -b fix/<short-description>
```

Branch naming conventions:

| Prefix | When to use |
|---|---|
| `feature/` | New functionality, new Terraform resources, new app routes |
| `fix/` | Bug fixes, correcting broken configs or SQL |
| `chore/` | Dependency updates, tooling, CI changes |
| `docs/` | README, DISTRIBUTION.md, inline comment updates only |

---

## Commit Messages

Use [Conventional Commits](https://www.conventionalcommits.org/) format:

```
<type>: <short summary>

<optional body>
```

Common types: `feat`, `fix`, `chore`, `docs`, `refactor`

Examples:
```
feat: add explain plans collector to Alloy config
fix: rename db_o11y to db-o11y in monitoring SQL
docs: add bastion EC2 setup instructions to README
chore: add terraform.tfvars and setenv.sh to .gitignore
```

---

## What Not to Commit

The following are gitignored and must never be committed:

| File / Pattern | Reason |
|---|---|
| `.env`, `.env.*` | Contains DB credentials |
| `alloy/setenv.sh` | Contains Grafana Cloud API keys |
| `terraform.tfvars` | Contains Terraform secrets |
| `.terraform/` | Provider binaries ŌĆö regenerated by `terraform init` |
| `node_modules/` | Dependencies ŌĆö regenerated by `npm install` |

## Making Changes

### Terraform

- Run `terraform plan` before `terraform apply` and review the diff
- Do not modify `versions.tf` provider constraints without discussion
- `rds.force_ssl = 1` is required ŌĆö do not disable SSL
- Init scripts (`init/01-seed.sql`, `init/02-monitoring.sql`) run automatically on provision; test SQL changes manually via psql before committing

### Alloy (`alloy/config.alloy`)

- Both DSN strings must use `sslmode=require` ŌĆö RDS rejects unencrypted connections
- The port must match the SSM tunnel local port (`5433` when using the tunnel, `5432` for direct connections)
- Use `host.docker.internal` when running Alloy in Docker on macOS

### App (backend / frontend)

- Backend changes: test API endpoints manually or with k6 before opening a PR
- Frontend changes: verify with `npm run dev` against a live backend
- Do not commit `app/backend/.env` ŌĆö use `.env.example` as the template

---

## Pull Requests

1. Push your branch to the remote:
   ```bash
   git push -u origin feature/<short-description>
   ```
2. Open a PR against `main`
3. PR title must follow Conventional Commits format (e.g. `feat: add postgres slow query dashboard`)
4. Include in the PR description:
   - What changed and why
   - Any manual testing steps taken
   - Whether a `terraform apply` or reboot is required

---

## AWS Credentials

- Use the `grafana-se-<yourInitials>` profile for SSM tunnels and general access
- Use the `terraform-rds` profile (assuming `TerraformRDSRole`) for Terraform operations
- Never hardcode credentials in any file ŌĆö use `aws configure --profile` and environment variables
